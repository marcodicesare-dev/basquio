## Problem 1: "Review suggested" in red destroys trust (CRITICAL UX)

### What the user sees

Big red "Review suggested" in the STATUS card — the most prominent visual signal on the page. Screams "something is wrong" to a user who just waited 20+ minutes and paid 29 credits.

### What actually happened

**QA PASSED.** Verified from DB:
```
artifact_manifests_v2.qa_passed: true
artifact_manifests_v2.qa_report.tier: "green"
artifact_manifests_v2.qa_report.issues: [] (empty)
deck_runs.delivery_status: "reviewed"
```

The red label comes from `run-progress-view.tsx:739`:
```typescript
const reviewSuggested = snapshot.summary?.qaPassed === false || (snapshot.qualityWarnings?.length ?? 0) > 0;
```

`qaPassed` is `true`. But `qualityWarnings` has 10 entries because `collectQualityWarnings()` (`jobs/[jobId]/route.ts:96-115`) merges:
- `cost_telemetry.phases.finalLint.actionableIssues` — 1 lint note
- `cost_telemetry.phases.publishDecision.advisories` — 9 advisory notes

These are INTERNAL QA observations:
- "Slide 16 writing issue: Text appears to be it but the deck language is en (title)"
- "Rendered slide 6 advisory underfilled_cards: Left evidence card shows only a single KPI metric"
- "Rendered slide 8 advisory chart_dead_space: Fee band chart occupies only ~60% of the frame width"

Professional-grade QA notes useful for internal review. TERRIFYING to a paying customer.

### Fix

```typescript
// run-progress-view.tsx:739 — ONLY show red status when QA actually failed:
const reviewSuggested = snapshot.summary?.qaPassed === false;
```

**Status card values:**
- `qaPassed === true` → **"Ready"** (green) — always, regardless of advisory count
- `qaPassed === false` → **"Review suggested"** (amber, not red)
- Failed → **"Failed"** (red)

Move advisories behind a collapsible "QA notes" at the bottom of the page — not in the hero.

### Files to modify

| File | Line | Change |
|------|------|--------|
| `apps/web/src/components/run-progress-view.tsx` | 739 | Remove `qualityWarnings.length > 0` from condition |
| `apps/web/src/components/run-progress-view.tsx` | 835-836 | Green "Ready" when qaPassed, amber when not |
| `apps/web/src/components/run-progress-view.tsx` | 815-819 | Remove advisory dump from hero; add collapsible QA notes section at page bottom |

## Problem 2: Wall of text above stats cards

### What the user sees

Two dense paragraphs of internal jargon:

> "FELFEL Slide Library.pptx parsed cleanly and is available to guide layout and theme choices for this run. This reflects template interpretation status, not a final fidelity score."

> "Basquio published the artifacts successfully, but flagged review notes such as: Slide 16 writing issue: Text appears to be it but the deck language is en (title) lint:Slide 16 writing issue..."

### Root cause

1. **Template summary** (`run-progress-view.tsx:812-819`): `templateSummary.detail` shown unconditionally. When template parsed perfectly (`status: "parsed_successfully"`, 0 warnings), this text adds zero value.

2. **Quality warnings** (`run-progress-view.tsx:815-819`): First 2 items from raw `qualityWarnings[]` dumped with machine-readable prefixes ("lint:", "contract:", "advisory underfilled_cards:").

### Fix

1. **Template summary:** Only show if there's a warning or fallback. Success = show nothing (or a small "FELFEL template applied" pill).
2. **Quality warnings:** Remove entirely from hero section. If retained anywhere, strip prefixes and put behind expandable.

## Problem 3: Preview thumbnails are empty gray boxes

### What the user sees

Three gray rectangles with "Slide 1", "Slide 3", "Slide 18" — no visible content.

### What's in the DB

3 preview PNGs exist, ~9,840 bytes each, stored in Supabase `artifacts` bucket.

### Root cause

Previews are **SVG-to-PNG renders of slide METADATA**, not actual slide screenshots. Generated by `buildSlidePreviewSvg()` (`generate-deck.ts:7157-7204`) which draws a stylized card with title/subtitle/bullets from the manifest.

At 9,840 bytes for a 1200x675 PNG, these contain minimal visual content — they're abstract placeholder cards, not slide thumbnails. The user expects to see what their actual slides look like.

### Fix

**Immediate:** If preview PNGs contain no meaningful visual content (< 5KB and no chart data), hide the preview section entirely. Empty previews are worse than no previews.

**Short term:** Improve SVG template — use template brand colors, show chart type icons, render title text prominently.

**Long term:** Render actual slide thumbnails from the PPTX using LibreOffice headless or a rendering API.

## Problem 4: Logo IS injected but is INVISIBLE (27x31 pixels)

### What the user expects

Their FELFEL logo prominently displayed on the cover slide.

### What actually happens — verified from PPTX binary analysis

The logo IS correctly extracted, stored, and injected. The entire pipeline works:

```
Template profile DB (template_profiles.template_profile):
  brandTokens.logo.imageBase64: PRESENT (502 chars, data:image/png;base64,...)
  brandTokens.logo.position: { x: 0.068, y: 1.087, w: 0.305, h: 0.35 }
  brandTokens.injection.logoBase64: PRESENT (same data)
  brandTokens.injection.logoPosition: { x: 0.068, y: 1.087, w: 0.305, h: 0.35 }

Generated PPTX (Deck Ready.pptx):
  ppt/media/client_logo.png: 358 bytes, 27x31 pixels ← INJECTED CORRECTLY
  ppt/slideMasters/slideMaster1.xml: <p:pic desc="Client Logo"> ← PRESENT
  Position: (0.07, 1.09) inches, size: 0.30 x 0.35 inches

Original FELFEL template (FELFEL Slide Library-v1.pptx):
  ppt/media/image2.png: 358 bytes, 27x31 pixels ← IDENTICAL FILE
  Used in 7 slide layouts as a tiny "Symbol" brand accent mark
```

**The logo is there. It's just 27x31 pixels at 0.30x0.35 inches — literally a dot.**

The FELFEL template uses this as a tiny decorative symbol mark on slide layouts. It was never meant to be the primary logo. The template engine found it, preserved its original size, and injected it faithfully. But a 0.30-inch image on a 13.33-inch-wide slide is invisible.

### Additional finding: Aspect ratio mismatch

```
FELFEL template: 10.83 x 7.50 inches (4:3 standard)
Generated deck:  13.33 x 7.50 inches (16:9 widescreen)
```

The generator uses 16:9 regardless of the template's aspect ratio. Logo position happens to still work (it's in the corner), but any proportional layout from the template would be distorted.

### Fix

**1. Enforce minimum visible logo size during injection** (`packages/render-pptx/src/apply-template-branding.ts`):

```typescript
const MIN_LOGO_DISPLAY_WIDTH = 1.2; // inches — visible at any zoom
const scale = Math.max(1, MIN_LOGO_DISPLAY_WIDTH / injection.logoPosition.w);
const displayW = injection.logoPosition.w * scale;
const displayH = injection.logoPosition.h * scale;
```

**2. Reposition logo for cover slides:**

Instead of (0.07, 1.09) — a layout accent position — use a standard cover logo position:
- Top-left: (0.5, 0.4) at scaled size
- This makes the logo visible and recognizable

**3. Match template aspect ratio in generated deck:**

Read `<p:sldSz>` from the template's `ppt/presentation.xml` and pass through to PptxGenJS `layout` option. A 4:3 template should produce a 4:3 deck.

**4. Long-term: Ask user to upload logo separately.**

Auto-detecting logos from PPTX internals is inherently fragile. A dedicated logo upload during template import would guarantee the right image at the right size.

### Files to modify

| File | Change |
|------|--------|
| `packages/render-pptx/src/apply-template-branding.ts` | Enforce minimum logo size (1.2in width), reposition for cover |
| `packages/template-engine/src/index.ts` | Extract slide dimensions from template for aspect ratio matching |
| `packages/workflows/src/generate-deck.ts` | Pass template aspect ratio to PptxGenJS layout |
