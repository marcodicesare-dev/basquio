# Working rules for Claude sessions on Basquio

**Version:** 1.0
**Locked:** 2026-04-19
**Load on every session.** These rules are non-negotiable. If a session output violates any of them, reject the output and redo.

These rules are short on purpose. Read them at the start, keep them in mind, do not write walls of text that drown the signal.

---

## 1. Research before strategy. Always.

Before writing any strategic recommendation, verify the claim with a source. If you do not have a source, say "I do not know, I need to research it." Do not pattern-match to a plausible answer.

Tools to use: Firecrawl CLI (`firecrawl search`, `firecrawl scrape`) with the authenticated key, WebFetch, Supabase direct queries. Agents can be used for parallel research but not as a substitute for direct verification on critical claims.

Every factual claim in any output needs a URL or a verified in-repo reference.

## 2. Spec first, build after approval

When asked to build, write the spec. Do not implement. Marco reviews the spec, approves, then a separate session builds it. This prevents wasted work and keeps the scope honest.

Specs include: user flow, acceptance criteria, technical approach, files touched, timeline calibrated to Marco's actual shipping pace (hours to days, not weeks).

## 3. No AI slop, ever

Banned words and phrases in any Basquio surface (product, website, email, spec, Discord message, commit):

- "dive deep" / "deep dive"
- "unlock"
- "leverage"
- "solutions"
- "empower"
- "elevate"
- "seamless" / "seamlessly"
- "game-changer"
- "revolutionize" / "revolutionary"
- "cutting-edge"
- "next-generation"
- "AI-powered" (as a standalone claim, no detail)
- "transform your workflow"
- Any sentence that could be generated by ChatGPT in 2022 with zero prompt

Replace with plain language. What does it actually do. Who does it help. What problem does it solve.

## 4. No em dashes

Not in code, not in copy, not in docs, not in Discord messages. Use:
- a period to end a thought
- parentheses for an aside
- a colon to introduce a list
- a comma and "and" to join clauses

Em dashes are an AI tell. Removing them is trivial. Do it.

## 5. Copy is human-crafted, short, immediate

Every piece of copy (button labels, headers, empty states, error messages, email subjects, marketing page, Discord message) must pass this test: "Would a specific real person say this out loud in a work conversation?"

If no, rewrite. Copy should be:
- sentence case (not Title Case On Every Word)
- active voice, present tense
- says what it does or what the user gets
- no jargon the user has not already encountered
- never "click here to learn more." Inline the info.

Reference: existing marketing copy in `/apps/web/src/content/` and `/docs/brand-identity-for-collaborators.md`. New copy matches that voice.

## 6. Production-grade visuals every session

Never ship placeholder design. Never ship "we will polish later." Every UI state (including empty states, error states, loading states) is designed to the same bar as the happy path.

Visual rigor baseline:
- Use tokens from `/docs/design-tokens.md` only. Do not invent hex codes, font sizes, spacing values.
- Golden-ratio proportion for layout (1 : 1.618)
- Fibonacci spacing scale: 4, 8, 13, 21, 34, 55, 89 px
- Modular type scale 1.25: 13 → 16 → 20 → 25 → 32 → 40 → 50 → 64 px
- Line-height: 1.5 body, 1.3 subheads, 1.15 display
- Max reading line length: 66 characters
- Radius, shadow, border: from tokens only
- Brand palette only. Never introduce a new accent color.

Test: if a person who charges 2000 euro per hour for design saw this screen, would they ship it? If no, keep polishing.

## 7. Sub-50ms interaction feedback

Every user input (click, type, submit) shows visual feedback in under 50 ms. If a write operation takes longer than 50 ms, apply optimistic UI: the UI changes immediately, the network request confirms asynchronously, and a toast or silent reconcile handles failures.

Never ship a button that "thinks" for 300 ms before doing anything visible.

## 8. Self-serve UX

Every screen must answer: "what am I looking at" and "what do I do next" in under 3 seconds. Empty states teach the flow. Errors tell the user how to fix, not what went wrong. Primary action is always visible without scroll.

If a screen requires a human to explain it to a new user, redesign it.

## 9. No walls of text

Structure everything. Headings, bullets, tables, short paragraphs. A wall of prose is almost always a signal that the author has not finished thinking.

In final responses: lead with the answer, then structure the reasoning. Not the other way around.

## 10. Never fabricate status

If Marco asks "is X shipped?" and you are not sure, check the code. Run `git show`, read the file, query Supabase, run the pipeline. Never answer from memory of earlier session state. Code changes fast here.

## 10b. Provider contract changes need live smoke

If a change touches Anthropic tool wiring, `packages/workflows/src/anthropic-execution-contract.ts`, `buildClaudeTools()`, or the author or revise request envelope, run a live provider smoke on the exact path you changed.

Minimum rule:
- run `pnpm test:code-exec-no-webfetch` before claiming the change is safe
- run `pnpm exec tsx scripts/test-anthropic-skills-contract.ts --web-fetch-mode enrich` when the Sonnet or Opus enrich branch is still part of the release path
- if the change affects another contract branch, run the closest matching live smoke too
- unit tests are necessary, not sufficient
- if a historical spec or test says the provider allows something and the live API disagrees, the live API wins and the docs must be corrected in the same change

## 11. Respect shipping pace

Marco ships in hours. A task you would estimate at 3 weeks for a funded team ships here in 2 to 3 focused sessions. Calibrate timelines accordingly. Do not pad estimates. Do not recommend a 3-month multi-tenant lift when the foundation is already multi-tenant and the remaining work is a few database columns.

## 12b. No emojis. Ever.

Locked 2026-04-20 by Marco. Absolute ban on emojis in any Basquio surface: product UI, chat render, tool-call chips, status pills, empty states, website copy, email, Discord messages, spec examples, commit messages, PR titles, docs, memory entries.

The only allowed iconography is crafted SVG icons from Heroicons or Phosphor Icons. `apps/web` already has `@phosphor-icons/react` installed. Prefer Phosphor for consistency with the existing sidebar icons.

When uncertain which icon fits, scrape the library via Firecrawl (`firecrawl scrape https://phosphoricons.com/` or `firecrawl scrape https://heroicons.com/`) and pick the right one for the semantic meaning.

Common replacements:
- 🧠 (brain) → Phosphor `<Brain />` or `<Sparkle />`
- 📊 (chart) → Phosphor `<ChartBar />` or Heroicons `<ChartBarIcon />`
- ✨ (sparkle) → Phosphor `<Sparkle />` or Heroicons `<SparklesIcon />`
- ⚡ (bolt) → Phosphor `<Lightning />` or Heroicons `<BoltIcon />`
- ✅ (check) → Phosphor `<Check />` or Heroicons `<CheckIcon />`
- ❌ (cross) → Phosphor `<X />` or Heroicons `<XMarkIcon />`

Why: emojis are a low-effort AI tell. Basquio surfaces must look like a paid 2k-euro-per-hour design studio shipped them. Emojis signal "ChatGPT with extra steps," which is the exact positioning Basquio differentiates against.

Enforcement: grep for emoji codepoints in every commit. Reject any PR that introduces an emoji outside data fixtures or verbatim third-party quotes.

## 12. Strategic alignment is locked

Canonical strategy lives in:
- `/docs/strategy-basquio-motions.md`
- `/docs/motion1-gtm-playbook.md`
- `/docs/motion2-workspace-architecture.md`

Before proposing any strategic change, read those three files. If the change is inside the existing frame, edit the doc. If the change breaks the frame, surface it explicitly and let Marco decide before writing code.

Do not re-debate Crosby vs Legora. Do not re-anchor Motion 2 pricing to 30k per year. Do not confuse individual-user acquisition (Motion 3 PLG tail) with Motion 1 or 2. See the canonical doc sections for flip-flop prevention.
