## 1. The strategic shift the team locked in

Read the transcript line by line. The team converged on one product reframe:

> "Il lavoro più difficile è capire cosa vuole la gente... se ci vogliamo differenziare ancora di più da Claude è il far sì che la gente nel proprio account di Basquio abbia tutta questa memoria in modo più concreto." (Fra)

> "Una volta che ti sei abituato non te ne vai più perché dici: cazzo, questo è tutta la mia memoria, mi conosce. Di fatto il mio AI è tipo una sorta di mio ruolo copiato in un AI che mi aiuta a ricordarmi tutto."

This is a hard pivot in the JTBD framing:

| Before V1 | After this session |
|---|---|
| **data → output** generated by a super-specialised agent | **memory + context + data → output** generated by a super-specialised agent |
| Pitch: "upload data, get a finished deck" | Pitch: "Basquio remembers your clients, your stakeholders, your style. Ask anything, get an answer that already knows the context." |
| Differentiator: deck-quality + speed | Differentiator: persistent, scoped memory that compounds session over session |
| Distribution: Product Hunt, organic discovery | Distribution: 3-5 pre-built demo workspaces handed to target customers (BAS-174). PH deprioritised. |
| Output type: deck | Output type: any analyst deliverable (memo, narrative, chart spec, deck), produced from the workspace's accumulated context |

**Verified against canonical strategy** ([docs/strategy-basquio-motions.md](strategy-basquio-motions.md)): "zero CPG/FMCG incumbent uses memory/context as the primary pitch." This is the locked whitespace. V1 must lead with memory, not file upload.

**Verified against architecture lock** ([docs/motion2-workspace-architecture.md](motion2-workspace-architecture.md) §1): "the product is the workspace where the analyst lives, not the deck generator. The deck falls out when someone asks for a deck. The moat is cross-session + cross-client memory."

**The shipped V1 does not lead with memory. It leads with file upload. That is the central audit finding.**

## 3. UX audit, gap by gap

### 3.1 The empty state reinforces the old JTBD
[page.tsx:62-69](../apps/web/src/app/(workspace)/workspace/page.tsx)

> "Drop a file. Ask anything."

This is the data → output framing. The new framing is **memory-first**. The right empty state for the team's pivot is something like:

- **Headline:** "Set up what Basquio remembers about your work."
- **Lede:** "Tell it who you cover, which clients you serve, which metrics matter. Then upload anything. Every answer leans on what you taught it."
- **First action:** *not* a drop zone. A **scope picker / setup wizard** ("Which client are you starting with?"), then optional upload, then "Ask anything".

Drop-a-file as the first call to action recreates ChatGPT. The differentiator is the memory you built up *before* the upload.

### 3.2 The page handle is a developer hack, not a workspace identity
[page.tsx:94-96](../apps/web/src/app/(workspace)/workspace/page.tsx)

```tsx
{userEmail ? userEmail.split("@")[0].replace(/\./g, " ") : "your workspace"}
```

Renders as "marco" (capitalized email username). A workspace built around persistent memory needs a real workspace name (e.g. "Marco's CPG insights workspace" or per-client "Mulino Bianco workspace"). The current handle says nothing about the analyst's domain or scope coverage.

### 3.3 The stat row reads architecture, not value
[page.tsx:99-117](../apps/web/src/app/(workspace)/workspace/page.tsx)

> 9 files · 28 entities · 8 answers

These are storage counts. The user does not buy storage. Replace with value-led metrics:

- Clients you cover: 1
- Stakeholders Basquio knows: 6
- Categories tracked: 3
- Hours of context compounded: ~12
- Last deliverable: 2 days ago

The metrics should make the user feel that *the workspace knows things*, not that *the database has rows*.

### 3.4 The Timeline shows entities, not memory
[workspace-timeline.tsx](../apps/web/src/components/workspace-timeline.tsx)

The Timeline groups entities by type and lets the user click into mentions and facts. Useful, but it surfaces what the *extractor* found in *your uploads*. It does not surface:

- What Basquio has *learned* (procedural memory: "lead with the headline", "Elena reviews Mondays")
- What you *told it directly* (the Memory Tool, never exposed)
- What it *re-uses across sessions* (the compounding loop)

A user looking at the current Timeline cannot tell the difference between "I uploaded a deck and the AI extracted a name" and "the AI now knows Elena is Head of Category and reviews on Mondays". That distinction is the entire pitch.

### 3.5 Scopes are a dropdown, not navigation
[workspace-prompt.tsx:128-145](../apps/web/src/components/workspace-prompt.tsx)

Scopes (`workspace`, `analyst`, `client:Mulino Bianco`, `category:Snack Salati`) live in a `<select>` at the top of the prompt. In Harvey and Legora, scopes are a **left-rail navigation primitive**. Switching scope changes what the workspace shows, not just what the next prompt searches.

A scope sidebar would let the user think "I'm working on Mulino Bianco today" and have the entire workspace narrow to that client.

### 3.6 The procedural memory loop is invisible
[deliverables/[id]/route.ts:79-91](../apps/web/src/app/api/workspace/deliverables/[id]/route.ts)

When the user edits a deliverable and saves, the API writes a memory entry under `/memories/{scope}/preferences/edits-{id}.md`. The user never sees this. Next time they ask, the model uses it silently. The "I taught it once" feedback loop is the moat, and it is hidden behind a checkbox labelled "Remember this edit as a preference for next time" with no observable consequence.

This needs a visible "Basquio learned X from your edit" toast plus a place to inspect what was learned.

### 3.7 No way to teach Basquio anything that is not in a file
There is no UI to add a memory entry directly. The user cannot say "Elena reviews Mondays at 3pm" without uploading a transcript that mentions it. This is a major gap given the team's framing of "tutta la mia memoria". Much of analyst memory is heuristics and stakeholder context that never lives in a document.

### 3.8 Three competing primary actions on the right column
On the populated state ([page.tsx:143-173](../apps/web/src/app/(workspace)/workspace/page.tsx)) the right column has, in order: prompt, recent answers, add-to-workspace, files. Each renders as a section with the same visual weight. The user's eye has no anchor. The prompt should be unambiguously the hero; recent answers should be a peek; uploads and files should collapse behind a single secondary affordance.

### 3.9 Files vs Documents vs Sources naming is inconsistent
- Section title "Files"
- Inline drop zone "Add to the workspace"
- Empty state "Upload your first file"
- Suggestions panel "Summarize the key findings in {filename}"
- Citations refer to source labels `[s1]` mapping to filenames

Pick one term per concept and use it everywhere.

### 3.10 The redesign I shipped does not match the new strategy
This is the most important UX finding. The hero, page header, and section structure were designed under the old frame. They are technically improved over the prior version (topbar fixed, editorial spacing, real hierarchy) but they do not surface memory, scope, stakeholders, or the demo-workspace use case. **The next pass needs to redo the information architecture, not just the styling.**

## 4. Architecture audit

What the backend supports today but the frontend does not expose:

1. **Browseable memory tree.** `memory_entries` is keyed by `(organization_id, scope, path)`. The schema supports a filesystem view. No UI calls it.
2. **Direct memory write.** The Memory Tool already handles `create / str_replace / insert / delete / rename`. A simple "Teach Basquio" form could call these without involving the model at all.
3. **Fact pinning.** The `facts` table has `superseded_by`, `valid_from`, `valid_to`. The user could pin "this fact is current and authoritative" and the agent would always rank it. No UI.
4. **Cross-scope queries.** `assembleWorkspaceContext` accepts `scope`. The UI never lets the user merge two scopes ("everything Mulino Bianco AND Snack Salati"). Trivial backend extension.
5. **Per-deliverable provenance ledger.** `workspace_deliverables.metadata` already records `chunk_count`, `fact_count`, `entity_count`. A deliverable-detail panel could show "this answer used 4 facts from your Q4 deck, 2 facts from Elena's transcript, and 1 procedural preference you taught it on Apr 9". The data is sitting there.
6. **Workspace-level identity.** `BASQUIO_TEAM_ORG_ID` is hardcoded ([constants.ts](../apps/web/src/lib/workspace/constants.ts)). For the demo-workspace-for-outreach motion (BAS-174) to work, the workspace concept needs a row of its own with `name`, `description`, `owner`, `members`, `kind` (personal / client-demo / team).
7. **Read-only sharing token.** Required for the BAS-174 outreach motion. The architecture supports it (one extra column on a `workspace_shares` table). No UI, no endpoint.
8. **CPG-specific scaffolding.** The whitespace claim is "specialised for CPG/FMCG analysts." There is no domain prior in the backend: no KPI dictionary seeded, no retailer canonical list, no category hierarchy. Every workspace starts from zero. Veronica would not feel that Basquio is "her" tool on session one.

The architecture is correct in shape. The work is to expose it and to seed it with domain priors.

## 5. Strategy audit

### 5.1 BAS-174: demo workspace with pre-loaded context for outreach
The team's #1 distribution motion. Shipped V1 cannot fulfill it:
- No workspace concept of "demo workspace owned by Marco, viewable by prospect@kellanova.com"
- No share link
- No way to clone a workspace template ("Snack Salati category template")
- No domain priors to seed
- The seed script I wrote ([scripts/seed-workspace-marco.ts](../scripts/seed-workspace-marco.ts)) is the right shape for a one-off but not productised

**Strategic implication:** until BAS-174 is buildable, the team's outreach motion has no demo to hand over. The current site sends a prospect to "sign in, then upload your data, then maybe Basquio works." That fails on the first call.

### 5.2 BAS-175: 10-slide Sonnet Excel chart output
The deck output path lives in the existing (app) shell. The workspace generates memos. There is no "Generate deck" button on a workspace deliverable. The legacy "data → deck" use case (which still pays bills today) is orphaned from the workspace that the team just decided is the future.

**Strategic implication:** the bridge from "workspace knowledge → deck artifact" is the chokepoint that makes both motions cohere. Without it, the team has two products: legacy deck pipeline (revenue today) and workspace beta (future moat) with no flow between them.

### 5.3 PH deprioritisation
The team agreed PH is not the right channel right now. The shipped V1 has zero outreach-mode features (shareable workspace, prospect-specific seeding, "talk to Marco" CTA visible to a non-team viewer). PH would have failed regardless. Nothing to fix here, but **the redesign budget should now go to the demo-workspace and outreach surface, not to a public landing.**

### 5.4 Vertical specialisation claim
[docs/strategy-basquio-motions.md](strategy-basquio-motions.md) §1 claims Basquio is "specialised for CPG/FMCG analysts." The shipped UI uses zero domain language. No KPI vocabulary, no retailer references, no category structure visible. A prospect who lands on the workspace cannot tell it apart from a generic file-chat tool. The vertical claim is implicit in the founders' heads, explicit in the strategy doc, and absent from the product surface.

## 6. Next-pass priorities (no implementation here, just sequence)

In order of impact on the team's locked strategy:

### Priority 1: Make memory the hero
Add a left-rail showing scopes (workspace / analyst / each client / each category) and, for each scope, what Basquio remembers (count of facts, last update, last deliverable). Change the empty-state hero from "Drop a file. Ask anything." to a memory-led setup flow ("Tell Basquio what you cover, then upload anything"). Add a "Memory" page that browses the `/memories/{scope}/...` filesystem with view + edit + add-direct.

### Priority 2: Demo-workspace concept (BAS-174)
Promote `organization_id` from a constant to a real `workspaces` table. Add `kind: personal | client-demo | team`. Add a `workspace_shares` table for read-only share links. Build a "create from template" flow seeded with a category bundle (Snack Salati, Skincare, etc.). This unlocks Marco/Ale's outreach motion.

### Priority 3: Bridge workspace → deck (BAS-175)
Add a "Generate deck" button on every workspace deliverable that hands the answer + cited sources + workspace context to the existing Inngest deck pipeline. The workspace becomes the brief; the deck pipeline is the renderer. The two products merge.

### Priority 4: Provenance on every answer
On every deliverable, show "this answer used X facts, Y memory entries, Z source excerpts" as a real visible feature, with click-through to inspect each. Surfaces the moat the team just bet on.

### Priority 5: CPG scaffolding seed
Ship a default workspace bundle: KPI dictionary (value share, ROS, ND, WD, PSU, promo pressure), Italian retailer canonical list (Coop, Conad, Esselunga, Carrefour, Selex, Tigre), category hierarchy (the existing `niq-storymasters` knowledge), procedural memory for FMCG analyst conventions. Every new workspace inherits this. Veronica feels at home on minute one.

### What NOT to build in the next pass
- No further visual polish on the current page structure. The IA is wrong; restyling it is wasted.
- No new file types in the parser. Coverage is fine for V1.
- No public marketing surface. The strategy says outreach.
- No multi-tenant org migration yet. V1 still uses the single team org. Multi-tenant only when BAS-174 demo workspace forces it.

## 7. What I got wrong, explicitly

For the next agent that picks this up, the things I missed in the V1 build:

1. I read the spec ([docs/spec-v1-workspace.md](spec-v1-workspace.md)) literally and shipped its three screens. I did not re-read the strategic context in [docs/motion2-workspace-architecture.md](motion2-workspace-architecture.md) §1 ("the product is the workspace, not the deck generator. The moat is memory") and translate it into UX. I built a file-chat tool with a memory backend, not a memory-led workspace.
2. I treated the scope picker as a generic dropdown. It should have been the navigation spine.
3. I built procedural memory writeback in the API but never gave the user a window into it.
4. The redesign I shipped one hour ago applied editorial polish to the wrong information architecture. It looks better than what came before. It is still wrong for the strategy.
5. I did not build any of the BAS-174 outreach surface, even though the team session marked it as the top priority for the next 2 weeks.

The architecture is largely right. The UX is one full pass behind the strategy. The next session should re-spec the workspace home with memory as the protagonist, then build that. Avoid the temptation to keep iterating on the current shell.
